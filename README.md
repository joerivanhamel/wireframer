# Deterministic UX Architecture Engine

This repository contains the system specification for a deterministic, zero-trust UX Architecture engine. The engine operates as a strict, state-driven pipeline that maps target and competitor web pages to a standardized structural ontology. By analyzing multiple competitor pages, the engine identifies structural consensus and outputs a semantic HTML5 wireframe styled with custom CSS.

---

## System Architecture

The engine functions through a decoupled pipeline:

```
[Web URLs] ──> [Zero-Trust Verification] ──> [Consensus & Audit] ──> [JSON Blueprint] ──> [HTML5 Wireframe]
```

1. **Deterministic State Machine**: Every execution step is locked behind sequential states. The engine cannot bypass or fast-forward states without meeting predefined completion criteria.
2. **Controlled Vocabulary**: Layout layouts are restricted to exactly 14 predefined semantic structures to prevent categorization drift.
3. **Decoupled JSON-Schema**: Data representation is separated from the rendering layer. The engine generates a standardized `UX-Blueprint` JSON specification before synthesizing visual layout code.

---

## State Machine Execution Flow

The engine progresses through 6 explicit states. Every output generated by the engine must begin with an explicit indicator of its current state, formatted as **`[CURRENT STATE: StateName]`**.

### State Details

| State | Name | Input Requirement | Operational Process | Transition Criteria |
| :--- | :--- | :--- | :--- | :--- |
| **1A** | `TARGET_INIT` | Target URL (or "None") | Prompts the user for the primary URL targeted for redesign. | A valid URL or "None" is provided. |
| **1B** | `COMP_INIT` | 3 to 5 competitor URLs | Prompts the user for competitor baseline URLs of the same page type. | Exactly 3 to 5 valid URLs are provided. |
| **2** | `FETCH & VERIFY` | Web content retrieval | Executes a zero-trust audit to verify content retrieval. Outputs the exact `<title>`, `<h1>`, and a verbatim 5-word footer quote for each URL. | Hard verification evidence is confirmed by the user responding "Proceed". |
| **3** | `AUDIT` | Consensus mapping | Maps content to the Controlled Ontology, evaluates vertical placement, and outputs a consensus markdown table alongside a standardized JSON spec. | User provides `'APPROVE'` or submits an updated JSON block. |
| **4** | `SYNTHESIS` | HTML5 Compilation | Processes the JSON blueprint as the Absolute Single Source of Truth to output styled, semantic, and modular HTML5 wireframes. | Complete code blocks generated according to specifications. |
| **5** | `COMPLETE` | Process Termination | Prints a standard completion notification and stops execution. | Final state reached. |

## Controlled Vocabulary (Ontology)

Layout elements must fit into one of these 14 categories:

1. `HeroBanner` – Primary top-level value proposition.
2. `SocialProof` – Customer logos, raw user counts, trust badges, or press mentions.
3. `ProblemAgitation` – Explores user friction or pain points preceding the solution.
4. `FeatureGrid` – Modular matrix presenting benefits and technical capabilities.
5. `HowItWorks` – Multi-step onboarding, setup, or delivery progression.
6. `ProductComparison` – Feature matrices presenting a direct comparison of alternatives.
7. `PricingTable` – Standard cost tiers, packages, or plans.
8. `InteractiveTool` – Calculators, selector quizzes, or dynamic assessments.
9. `Testimonials` – Quotes, customer success reviews, or case study snippets.
10. `FAQ` – Accordion or toggle-based question-and-answer patterns.
11. `TrustFramework` – Compliance listings, security certifications (SOC2, ISO), or regulatory standards.
12. `ResourceHub` – Secondary resource links, blog articles, downloads, or lead-generation blocks.
13. `CallToAction` – Highly focused, bottom-tier conversion modules.
14. `Footer` – Navigation maps, legal disclaimers, copyrights, or registration numbers.

## UX-Blueprint Schema

The JSON specification generated during **[STATE 3: AUDIT]** uses the following format:

```json
{
  "$schema": "https://ux-engine.org/schemas/blueprint.v1.json",
  "metadata": {
    "target_url": "string",
    "competitors": [
      "string"
    ]
  },
  "layout": {
    "vertical_order": [
      "string"
    ],
    "geometry_consensus": {
      "ontology_term_key": {
        "desktop_columns": 1,
        "element_alignment": "left | right | center",
        "cta_orientation": "stacked | horizontal-inline | none",
        "cta_primary_position": "left | right | center | none"
      }
    }
  }
}
```

## Synthesized HTML5 Design Standards

During **[STATE 4: SYNTHESIS]**, the engine generates HTML5 wireframes obeying these core design variables and constraints:

* **Card-Based Borders**: Layout boundaries use high-contrast dashed borders (`--border-style-dashed`) in grayscale shades to make structural hierarchies visually apparent.
* **Component Tags**: Modules render their ontological name inside a absolute-positioned corner tag (e.g., `<span class="wireframe-tag">herobanner</span>`). Custom ontological descriptors outside the standard classification are disallowed.
* **Semantic Placeholders**: Visual content is designated using a hybrid placeholder syntax that targets the copywriter's intent (e.g., `<h1>[H1 Headline: 24/7 Digital Fall Alarm with Automatic Detection]</h1>`).

## Usage Guidelines

To configure a target environment (such as an LLM orchestrator) to run this system, import this specification file and invoke the prompt definitions verbatim. Be sure to strictly follow the progressive confirmation checks under **[STATE 2: FETCH & VERIFY]** to maintain alignment with the deterministic guidelines of the engine.
